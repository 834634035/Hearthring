## 总体主线

这段记录不是单一主题，而是一条连续的前端底层学习路径。它从 Webpack 的 seal 阶段开始，逐步延伸到浏览器网络协议、HTTPS、JavaScript 执行机制、闭包、Event Loop、渲染流水线，最后进入 V8 的对象结构优化和 Inline Cache。

可以把整段学习路线概括为四层：

1. 构建层：Webpack 如何把 module 组织成 chunk，并最终生成 assets。
2. 网络层：HTTP/1.1、HTTP/2、HTTP/3、TCP、QUIC、HTTPS 如何影响资源加载。
3. 运行时层：JavaScript 同步任务、异步任务、宏任务、微任务、闭包与 Event Loop 如何协作。
4. 引擎层：V8 如何通过 Hidden Class、Dictionary Mode、Inline Cache、Stub 等机制优化对象属性访问。

## Webpack Seal 阶段

对话一开始围绕 Webpack 的 `seal` 阶段展开。核心理解是：`make` 阶段负责把模块依赖关系构建出来，而 `seal` 阶段负责把这些已经构建好的 module 封装、优化、编号，并转化为最终可输出的资源。

`seal` 阶段可以理解为 Webpack 从“模块依赖图”走向“最终资源图”的关键阶段。它会根据入口、动态导入、运行时需求等信息生成 chunk graph，把 module 分配到不同 chunk 中，然后执行优化、分配 id、生成代码、生成 assets。

一个简化流程是：

```text
module graph
  -> chunk graph
  -> optimize modules/chunks
  -> assign module ids / chunk ids
  -> code generation
  -> create assets
  -> seal finished
```

这里需要注意，`seal` 结束的标志不是“文件已经写入磁盘”，而是 compilation 内部的 assets 已经生成完成。真正把 assets 写到输出目录，是后续 `emit` 阶段做的事情。

## Chunk、Assets 与缓存

记录中多次讨论了为什么构建工具要拆分 chunk，以及为什么不是文件越少越好，也不是文件越多越好。

在 HTTP/1.1 时代，浏览器受连接数量、请求排队、握手成本等限制影响较大，所以前端工程倾向于把很多小文件打包成较少的大文件，减少请求数量。

到了 HTTP/2 以后，多路复用降低了大量小请求的连接成本，构建工具可以更积极地拆分 chunk。这样做的好处是缓存粒度更细：用户更新某个页面时，不必重新下载整个 bundle，只需要下载变化的 chunk。

但 chunk 也不能无限拆。拆得太细会带来额外的请求调度、解析、编译和缓存管理成本。因此工程上的目标不是“越拆越好”，而是根据入口、路由、动态导入、公共依赖和缓存收益来控制拆分粒度。

## HTTP/1.1、HTTP/2 与 HTTP/3

网络部分的核心问题是：为什么 HTTP 协议不断演进，以及不同版本分别解决了什么瓶颈。

HTTP/1.1 的关键改进是 Keep-Alive。它允许一个 TCP 连接复用多次请求，避免每个资源都重新建立 TCP 连接。但 HTTP/1.1 在同一个连接上仍然存在应用层队头阻塞：前一个响应没有完成，后面的响应通常无法越过它先返回。

HTTP/2 的核心能力是二进制分帧和多路复用。它把请求和响应拆成带有 stream id 的 frame，使多个请求可以在同一个 TCP 连接中交错传输，再由浏览器按 stream id 重新组装。它还引入了 Header 压缩，减少重复请求头带来的传输浪费。

但 HTTP/2 没有解决 TCP 层队头阻塞。因为它仍然运行在 TCP 上，而 TCP 必须按字节序可靠交付。一旦某个 TCP 包丢失，即使后续包已经到达，应用层也不能提前读取后续数据。

HTTP/3 的关键变化是基于 QUIC，而 QUIC 基于 UDP 实现了可靠传输、多路复用、连接迁移和更快的建连。QUIC 的多个 stream 在传输层相互独立，某个 stream 丢包时，不会阻塞其他 stream 的交付。

## 应用层队头阻塞与 TCP 层队头阻塞

记录中特别强调了这两个概念不能混在一起。

应用层队头阻塞是协议使用方式导致的。例如 HTTP/1.1 在一个 TCP 连接上串行处理多个响应时，如果第一个响应很大，第二个响应即使已经准备好，也要等第一个响应结束。

TCP 层队头阻塞是可靠有序字节流导致的。只要某个较早的 TCP 包丢失，后面的字节即使已经到达，也不能交给上层应用使用，必须等待丢失包重传。

一句话区分：

```text
应用层队头阻塞：应用协议要求你排队。
TCP 层队头阻塞：传输层要求字节按顺序交付。
```

HTTP/2 解决了 HTTP/1.1 的应用层队头阻塞，但没有解决 TCP 层队头阻塞。HTTP/3 通过 QUIC 才真正改善了传输层多路复用下的阻塞问题。

## HTTPS、RSA 与 AES

记录中还讨论了 HTTPS 的安全基础：为什么公钥可以公开，为什么仍然能抵抗中间人攻击，以及 RSA 和 AES 的分工。

核心理解是：公钥公开并不危险，因为公钥主要用于加密或验签，私钥才是不能泄露的关键。攻击者知道公钥，并不意味着能解密只有私钥才能解开的内容。

HTTPS 实际使用的是混合加密。非对称加密适合解决身份认证和密钥交换问题，但性能较低；对称加密性能高，适合加密大量业务数据。因此 TLS 握手阶段会通过非对称加密或密钥协商建立会话密钥，之后用 AES 这类对称加密算法传输数据。

## JavaScript 执行上下文与闭包

中段学习转向 JavaScript 运行机制，重点讨论了执行上下文、词法环境、作用域链和闭包。

闭包不能只理解成“函数引用了外部变量”。更准确地说，闭包背后依赖词法环境。当函数被创建时，它能访问的外部词法环境已经确定。即使外层函数执行上下文已经出栈，只要内部函数仍然引用某些变量，引擎就必须让这些被引用的变量继续存活。

记录中还讨论了一个关键问题：如果闭包只用到了变量 `a`，那么同一作用域里的 `b` 和 `c` 是否也一定会被保留。这个问题引出了引擎优化：规范层面只规定可观察行为，引擎实现可以在不改变语义的前提下做变量裁剪，不一定机械地保留所有局部变量。

## Event Loop、宏任务、微任务与渲染

Event Loop 部分的主线是把同步任务、异步任务和浏览器渲染串起来理解。

一次点击事件可以看成一个完整 task。这个 task 中执行同步 JS，遇到 Promise 的 `.then` 会把回调放入 Microtask Queue。当前 task 的同步代码执行完后，浏览器会清空所有微任务。只有当当前 task 和其产生的微任务都执行完，浏览器才可能获得一次渲染机会。

因此，如果代码是：

```js
button.onclick = () => {
  div.style.width = "100px";

  Promise.resolve().then(() => {
    div.style.height = "100px";
  });
};
```

通常不会先渲染 width，再渲染 height。因为 width 的修改发生在当前 task 中，height 的修改发生在同一轮 task 之后的微任务中。微任务清空后，浏览器才可能统一执行一次渲染。

这解释了为什么浏览器不会每执行一行 JS 就渲染一次。频繁渲染会导致大量无意义的 layout、paint、raster、composite，并让用户看到闪烁和中间状态。Event Loop 给浏览器提供了一个批处理机制，让 DOM 修改可以在合适的时机合并成一次有效渲染。

## 渲染流水线

记录中多次提到 `layout -> paint -> raster -> composite`。

可以这样理解：

- layout：计算元素几何位置和尺寸。
- paint：生成绘制指令，描述元素应该如何画出来。
- raster：把绘制指令栅格化成像素。
- composite：把不同图层合成到屏幕上。

不是所有样式变化都会触发完整流水线。例如只改变 `transform` 或 `opacity`，在某些情况下可以绕过 layout 和 paint，直接走 compositor 相关流程。这也是前端动画常推荐使用 transform/opacity 的原因。

## V8 Hidden Class 与 Dictionary Mode

后段学习进入 V8 对象模型。

Hidden Class 在 V8 源码中通常称为 Map。它是 V8 为结构相同的对象创建的共享元数据，用来描述对象有哪些属性、属性顺序和属性布局。它不是对象本身，也不是属性值，而是对象结构的描述信息。

当对象按照稳定顺序添加属性时，V8 会沿着 transition tree 生成或复用 Hidden Class。例如：

```js
const user = {};
user.name = "Niko";
user.age = 18;
```

可以理解为：

```text
Map0: {}
  -> Map1: { name }
    -> Map2: { name, age }
```

如果大量对象都以相同顺序添加属性，它们就可以共享同一个 Map，属性访问也更容易被优化。

Dictionary Mode 则是另一种属性存储模式。当对象结构变得非常动态，例如频繁新增、删除属性，或者属性数量很多时，继续维护稳定 Hidden Class 的收益会下降。此时 V8 可能把对象切换到类似字典/哈希表的模式，更适合动态变化，但属性访问速度通常不如 Fast Properties。

所以不是简单地说 V8 有两套互斥对象系统，而是：V8 会根据对象结构是否稳定，在更适合快速布局访问的 Fast Properties 和更适合动态变化的 Dictionary Mode 之间选择。

## Inline Cache、Stub 与 Offset

Inline Cache 简称 IC，是 V8 为某一个属性访问点建立的运行时缓存机制。它属于代码位置，不属于对象。

例如：

```js
function printAge(user) {
  return user.age;
}
```

这里产生 IC 的位置是 `user.age` 这个访问点。IC 缓存的不是 `user` 这个对象，也不是 `age` 的属性值，而是：当这个访问点再次遇到某种 Hidden Class 的对象时，应该使用哪一种已经验证过的快速访问策略。

更准确地说，IC 通常会记录 Map 与访问 Stub 的关系。Stub 是一段优化好的小程序，用来快速完成属性读取或写入。Offset 只是这个访问策略中的一部分。

Offset 可以通俗理解为“从对象某个起点出发，偏移多少位置可以找到目标属性”。它不是绝对内存地址，而是相对于对象内部存储布局的位置描述。

所以不能把 Inline Cache 简化成“缓存 offset”。更准确的关系是：

```text
Hidden Class / Map：描述对象结构。
Offset：描述属性在该结构下的相对存储位置。
Stub：执行快速访问的小段优化代码。
Inline Cache：在某个代码访问点缓存 Map -> Stub 的关系。
```

## 为什么 Hidden Class 不能替代 Inline Cache

Hidden Class 回答的是“这个对象是什么结构”。Inline Cache 回答的是“对于这段代码来说，如果再次遇到这种结构的对象，我应该怎么最快访问它”。

这两个问题属于不同维度。一个是对象结构维度，一个是代码执行维度。

同一个对象属性可能出现在很多不同代码位置，每个访问点观察到的运行时对象类型也可能不同。因此 IC 必须属于具体的属性访问点，而不是只存在于 Hidden Class 上。

## IC 状态演化

记录最后引出了后续学习路线：IC 会根据某个访问点遇到的对象结构数量，从单态走向多态，甚至进入超多态。

- Monomorphic IC：某个访问点基本只遇到一种 Hidden Class。
- Polymorphic IC：某个访问点遇到少量几种 Hidden Class。
- Megamorphic IC：某个访问点遇到太多不同结构，优化收益下降。

稳定对象结构和稳定调用形态有利于 V8 保持单态 IC，从而让后续 JIT 优化更容易发生。

这段记录里还明确形成了一个输出偏好：技术学习时应该优先使用连贯自然段，而不是一两个字就换行、频繁冒号分段的表达。

后续笔记和讲解适合采用这种结构：

1. 先给严格定义，避免在定义上使用不准确的简化说法。
2. 再解释设计目标和问题背景。
3. 接着推导实现原理，区分规范要求和引擎实现。
4. 最后再使用类比，且明确类比只是帮助理解，不是真实实现。

这非常适合学习 Webpack、浏览器网络、Event Loop、V8 这类底层主题，因为这些知识最容易因为过度简化而产生误解。
